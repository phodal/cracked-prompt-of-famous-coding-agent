# Qodercli 架构设计指导文档

## 1. 文档目的

本文用于指导在 Qodercli 的 multi-agent 与 spec 机制下进行架构设计。目标不是写出“漂亮文档”，而是写出能被后续 agent 正确消费、评审、拆解、实现和验证的设计文档。

一句话原则：

架构设计必须既能让人理解，也能让后续 agent 按阶段稳定执行。

## 2. 设计前必须接受的约束

在 Qodercli 里，架构设计不是自由发挥，而是受到工作流结构约束。

设计前必须默认以下事实：

1. 需求文档会先于设计存在。
2. 验证策略文档会在设计前或与设计并行被消费。
3. 高层设计要经过 review，而不是直接进入编码。
4. 设计最终会被拆成详细设计任务，而不是直接靠对话执行。
5. 详细设计还会进入一轮 review。
6. 编码可能按任务串行执行并支持中断恢复。
7. 最终必须被 verifier 按预定策略验证。

这意味着你的设计如果不利于拆任务、不利于测试、不利于恢复，就不算合格设计。

## 3. 架构设计的核心目标

在这套机制里，一份好的架构设计至少要同时满足五个目标：

1. 正确承接需求。
2. 清楚表达模块边界。
3. 能拆成可执行任务。
4. 能被自动化验证。
5. 能在 review 中逐项对照需求通过审查。

因此，设计文档的好坏不能只看逻辑是否自洽，还要看它是否“可编排”。

## 4. 推荐设计原则

### 4.1 需求映射优先

每个功能点都要能在设计中找到落点。推荐做法是先建立一张映射表：

| 需求 ID | 目标行为 | 设计模块 | 接口/流程 | 验证方式 |
|---------|----------|----------|-----------|----------|
| F1 | xxx | Module A | Flow A | UT-01, AV-01 |
| F2 | xxx | Module B | Flow B | IT-01 |

如果设计文档里没有这种映射关系，review 阶段就很容易出现“看上去都写了，但无法证明覆盖完整”的问题。

### 4.2 模块边界清晰优先

高层设计要先回答三个问题：

1. 哪些模块新增，哪些模块复用。
2. 每个模块的责任是什么。
3. 模块之间依赖方向是什么。

如果模块职责不清，后果会直接传导到 LLD：

- 无法准确分任务。
- 容易形成循环依赖。
- 单元测试边界不清晰。

### 4.3 测试可设计优先

设计时必须同时思考验证，而不是写完设计再补测试章节。至少需要明确：

1. 哪部分适合 UT。
2. 哪部分必须用 IT 才能覆盖。
3. 哪些用户可见行为需要 E2E 或主动验证。
4. 外部依赖怎么 Mock。
5. 哪些动作带副作用，需要用户确认后再验证。

### 4.4 实施可分解优先

如果一个设计无法被拆成多个清晰任务，它通常存在两个问题之一：

1. 设计仍然太抽象。
2. 设计边界划分不合理。

好的高层设计应该让 LLD 很自然地产生若干任务，每个任务都具备：

- 明确文件落点。
- 明确依赖关系。
- 明确边界条件。
- 明确测试策略。

### 4.5 恢复友好优先

由于这套系统支持从 artifacts 恢复，因此设计要天然支持“中途停下再继续”。

具体要求：

- 任务顺序要清晰。
- 状态切换要可追踪。
- 不要设计必须一口气完成的大爆炸修改。
- 尽量把高风险改动拆成阶段性交付。

## 5. 推荐的高层设计结构

建议高层设计文档至少包含以下部分。

### 5.1 需求摘要

用很短的篇幅重述本次需求的目标、边界和假设，不要复写整份需求文档。

### 5.2 现有系统分析

这一节不是背景介绍，而是为了回答两个问题：

1. 现在已有哪部分可以复用。
2. 本次变更会切到哪些真实代码边界。

建议写法：

- 相关模块列表。
- 现有模式与约束。
- 需要保持兼容的接口或行为。

### 5.3 总体方案

这里应明确：

- 方案总思路。
- 关键流程变化。
- 模块关系。
- 新旧模块交互方式。

注意不要把这里写成实现伪代码。它应该描述结构与数据流，而不是具体语句。

### 5.4 模块设计

建议每个模块都按统一格式描述：

| 模块 | 类型 | 职责 | 输入 | 输出 | 依赖 | 风险 |
|------|------|------|------|------|------|------|
| Parser | New | 解析 xxx | raw input | parsed model | config loader | medium |

### 5.5 数据模型与接口

只写必要的结构和接口，不写实现。推荐内容：

- 关键实体。
- 请求/响应模型。
- 核心接口签名。
- 错误处理原则。

### 5.6 文件变更计划

高层设计里必须尽早标出会改哪些文件，原因是什么。推荐使用表格：

| 文件路径 | 动作 | 原因 | 风险 |
|----------|------|------|------|
| src/a.ts | modify | 接入新流程 | medium |
| src/b.ts | add | 新增适配层 | low |

这一步会显著帮助后续 LLD 做任务划分。

### 5.7 测试与验证计划

高层设计必须和验证策略文件呼应，至少明确：

- 对应哪些测试类型。
- 验证入口在哪里。
- 哪些行为可自动验证。
- 哪些行为依赖环境。

### 5.8 实施顺序与风险

建议写出：

1. 首先实现什么。
2. 哪些步骤有依赖。
3. 哪里最容易失败。
4. 如果需要回滚，应该以什么粒度回滚。

## 6. 推荐的详细设计结构

详细设计不是把高层设计扩写，而是把高层设计编译成任务。

### 6.1 任务总览文档

`overview.md` 建议包含：

- 任务列表。
- 依赖关系。
- 执行顺序。
- 状态列。
- 关键风险。
- 集成测试与 E2E 说明。

任务表建议固定字段：

| ID | Task Name | Type | File Path | Dependencies | Status | Design Doc |
|----|-----------|------|-----------|--------------|--------|------------|

### 6.2 单任务文档

每个 `task-NN-xxx.md` 应是独立可执行单元，建议固定包含：

1. Basic Information
2. Implementation Solution
3. Code Design
4. Boundary Conditions
5. Interaction with Other Modules
6. Unit Test Design
7. Mock Dependencies
8. Acceptance Criteria

核心标准不是内容多，而是 implementer 只读这一份文档就能开始干活。

## 7. 需求到设计到验证的映射方法

推荐采用下面这条链路做设计。

1. 从需求文档提取 feature points。
2. 为每个 feature point 指定模块承接点。
3. 为每个模块定义接口与数据流。
4. 把模块拆成任务。
5. 为每个任务定义单元测试。
6. 为跨模块行为定义 IT。
7. 为用户可见主路径定义 E2E 或主动验证。

可以直接用一张总表管理：

| Feature | Module | Task | UT | IT | E2E/AV |
|---------|--------|------|----|----|--------|
| F1 | A | task-01 | UT-01 | - | AV-01 |
| F2 | B+C | task-02,03 | UT-02 | IT-01 | E2E-01 |

这张表会显著提升 review 阶段的可审查性。

## 8. 设计时必须主动考虑的五类问题

### 8.1 边界条件

不要把边界条件留给 implementer 自己猜。设计中至少要覆盖：

- 空输入。
- 非法输入。
- 缺失配置。
- 依赖失败。
- 并发或重复执行。
- 回滚或幂等性要求。

### 8.2 兼容性

如果改动已有模块，设计要写清：

- 哪些旧接口不变。
- 哪些行为会变。
- 是否需要适配层。
- 是否需要灰度或 feature flag。

### 8.3 依赖方向

建议设计里显式写“谁依赖谁”，不要默认 reviewer 自己看得出来。原则很简单：

- 上层依赖抽象，不依赖实现。
- 新模块尽量向现有稳定接口挂接。
- 避免把控制流和数据流缠在一个模块里。

### 8.4 Mock 与可测试性

如果某个设计点不能被轻松 Mock，它大概率会在实现和验证阶段拖慢整个流程。设计时就应标出：

- 哪些外部资源需要抽象接口。
- 哪些地方需要注入依赖。
- 哪些状态需要观察点或日志。

### 8.5 主动验证入口

对于 CLI、API、文件生成这类功能，设计里要明确：

- 验证命令是什么。
- 工作目录是什么。
- 成功标准是什么。
- 是否有副作用。
- 清理动作是什么。

否则 `spec-verifier` 很难稳定执行主动验证。

## 9. 设计文档的写作要求

### 9.1 应该写什么

- 目标。
- 边界。
- 模块职责。
- 接口。
- 数据流。
- 文件变更计划。
- 测试与验证方案。
- 实施顺序。
- 风险与取舍。

### 9.2 不应该写什么

- 大段实现代码。
- 与本次变更无关的系统介绍。
- 模糊表述，比如“这里做一些处理”。
- 不能落地的泛化建议。
- 无法映射到模块、任务或测试的空洞原则。

### 9.3 推荐文风

推荐使用：

- 自然语言描述结构。
- 少量接口签名。
- 表格表达映射关系。
- 简单图示表达依赖与流程。

不推荐用大量伪代码代替设计说明。

## 10. 评审视角下的设计自检清单

在提交高层设计前，至少自检以下问题：

1. 每个需求点是否在设计里有明确承接位置。
2. 是否有模块职责重叠或空白区。
3. 是否存在不必要的新抽象或过度设计。
4. 是否能自然拆出详细设计任务。
5. 每个关键路径是否有测试映射。
6. 外部依赖是否具备 Mock 策略。
7. 变更文件列表是否明确。
8. 风险点是否可验证、可回滚、可恢复。

在提交详细设计前，再自检：

1. 每个任务是否足够独立。
2. 依赖关系是否正确。
3. 任务粒度是否合适。
4. 单元测试设计是否具体。
5. 实现者是否可以不看上下文直接执行该任务。

## 11. 推荐的设计输出模板

下面是一份适合 Qodercli 的高层设计骨架。

```markdown
# High-Level Design

## 1. Design Goal
[本次设计要解决的问题]

## 2. Requirement Mapping
| Req ID | Requirement | Design Entry | Verification |
|--------|-------------|--------------|--------------|

## 3. Existing System Analysis
- [可复用模块]
- [现有模式]
- [兼容性约束]

## 4. Proposed Architecture
### 4.1 Overview
[总体方案]

### 4.2 Module Responsibilities
| Module | Responsibility | Depends On | Notes |
|--------|----------------|------------|-------|

### 4.3 Data Flow
[关键流程说明]

## 5. Interface and Data Model
### 5.1 Core Interfaces
[仅接口签名]

### 5.2 Data Structures
[关键结构]

## 6. File Change Plan
| File Path | Action | Reason |
|-----------|--------|--------|

## 7. Test and Verification Plan
| Type | Scope | Entry |
|------|-------|-------|

## 8. Implementation Order
1. [步骤一]
2. [步骤二]

## 9. Risks and Trade-offs
- [风险]
- [取舍]
```

## 12. 适配 Qodercli 的设计方法总结

如果要把一句经验讲透，那就是：

在 Qodercli 里，好的架构设计不是“描述系统有多聪明”，而是“把系统拆成后续 agent 不会误解的结构”。

因此，最重要的不是写得多，而是让设计具备以下四个特征：

1. 可映射到需求。
2. 可拆分成任务。
3. 可被测试验证。
4. 可在中断后恢复执行。

只要设计同时满足这四点，它通常就能很好地适配这套 multi-agent spec 机制。
