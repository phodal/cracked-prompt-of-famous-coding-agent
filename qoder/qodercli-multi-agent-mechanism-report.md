# Qodercli Multi-Agent 与 Spec 机制分析报告

## 1. 报告目的

本文基于 `qodercli` 中抽取出的 prompt 与 skill 定义，分析其多 agent 协作机制，重点回答四个问题：

1. 这套系统如何把用户请求分发给不同 agent。
2. `spec` 工作流如何把需求、设计、实现、验证拆成多个阶段。
3. 为什么它要强制角色隔离、用户确认和文档化交付。
4. 这套机制对架构设计、可维护性、可恢复性和工程质量意味着什么。

这不是对单个 prompt 的逐段翻译，而是对整套机制的抽象和归纳。

## 2. 结论摘要

Qodercli 的核心不是“很多 agent”，而是“一个编排器 + 一组受限执行者 + 一套文档驱动状态机”。

它的本质机制有六条：

1. `spec-leader` 是唯一流程编排者，但不是实现者。
2. 需求、设计、实现、验证被拆成多个职责单一的 agent，且通过 prompt 明确禁止越权。
3. 每个阶段都要求输出 artifact，artifact 既是结果，也是下阶段输入。
4. 阶段转换不是自动进行，而是由用户在检查点显式确认。
5. 所有 sub-agent 都被视为“无记忆独立会话”，因此 prompt 必须显式携带完整上下文。
6. 整个系统不是一次性线性流水线，而是带评审回路与验证修复回路的有限状态机。

换句话说，这是一种“文档即状态、编排即控制、限制即质量”的 agent 工程框架。

## 3. 系统本质：不是聊天代理，而是受控工作流引擎

从 prompt 设计看，这套系统并不把 agent 当成自由协作者，而是当成不同阶段的专用执行单元：

- `spec-leader` 负责编排，不直接干活。
- `spec-requirement-analyser` 负责需求澄清和任务类型识别。
- `spec-verify-requirement` 负责把“要做什么”转成“如何验证”。
- `spec-hld-designer` 负责高层设计。
- `spec-lld-designer` 负责详细设计和任务分解。
- `spec-design-reviewer` 负责设计质量审查。
- `spec-implementer` 负责写代码和测试。
- `spec-verifier` 负责执行验证并生成失败报告。

因此，这不是传统意义上的“多智能体自由协作”，而是“严格工序化的代理流水线”。

## 4. 架构分层

### 4.1 编排层

编排层的核心是 `spec-leader`。它有三个关键特征：

1. 只负责调度，不负责直接分析代码、修改代码、执行测试。
2. 通过 `Task` 工具把完整上下文打包后分配给 sub-agent。
3. 必须在每次 sub-agent 执行后验证其产物是否完整，不完整就重试。

这意味着 `spec-leader` 不是“最强 agent”，而是“最受约束但拥有流程控制权的 agent”。

### 4.2 领域执行层

领域执行层由需求、设计、实现、验证等专用 agent 组成。这些 agent 的 prompt 都带有强约束：

- 哪些文件可以读。
- 哪些文件可以写。
- 是否允许执行改变系统状态的命令。
- 输出只能落到哪个目录。
- 完成后必须把控制权还给谁。

这种设计把 agent 从“会做很多事”的通用助手，变成“在局部规则内可靠工作的专职工人”。

### 4.3 工件层

真正把流程串起来的不是对话历史，而是 artifacts：

- `01-requirement.md`
- `01-verify-requirement.md`
- `02-high-level-design.md`
- `03-high-level-design-review-round-N.md`
- `04-low-level-design/overview.md`
- `04-low-level-design/task-NN-xxx.md`
- `05-low-level-design-review-round-N.md`
- 验证报告

artifact 在这里承担四个角色：

1. 阶段输出。
2. 下游输入。
3. 恢复点。
4. 质量审计证据。

这说明 Qodercli 的长期记忆主要不依赖模型上下文，而依赖文件系统中的结构化交付物。

## 5. 入口路由机制

`spec-leader` 对用户请求做三分法：

1. 闲聊或纯说明类请求：直接回复，不进入 spec 工作流。
2. 凡是与项目代码相关的正式任务：必须先交给 `spec-requirement-analyser`。
3. 明确要求恢复已有 spec：读取 artifacts 判断当前阶段，然后从中断点继续。

这里最重要的一条是：

只要不是闲聊，理论上都先进入需求分析器，而不是直接进入实现器。

这意味着系统把“需求识别”当作总入口网关，而不是把实现当成默认入口。

## 6. `spec` 工作流状态机

### 6.1 阶段拆分

完整流程可归纳为：

1. 需求分析。
2. 验证策略设计。
3. 用户确认是否进入设计。
4. 高层设计。
5. 高层设计评审。
6. 用户确认是否进入详细设计。
7. 详细设计。
8. 详细设计评审。
9. 用户确认是否进入编码。
10. 编码实现。
11. 自动化验证。
12. 失败修复循环。
13. 交付确认。

### 6.2 为什么要拆成这么多阶段

这不是为了流程好看，而是为了控制两个常见问题：

1. 大模型在需求不稳定时直接编码，返工成本高。
2. 大模型在跨阶段工作时容易混淆责任，出现“边设计边实现边测试”的失控行为。

阶段化让每一步只承担有限目标：

- 需求阶段解决“做什么”。
- 设计阶段解决“怎么做”。
- 实现阶段解决“把它写出来”。
- 验证阶段解决“它是否真的正确”。

### 6.3 三个用户检查点

系统强制设置三个 `Stage Transition Checkpoint`：

1. 需求完成后，用户决定进入设计、直接编码，或补充需求。
2. 高层设计评审通过后，用户决定进入详细设计、直接编码，或回改设计。
3. 详细设计评审通过后，用户决定进入编码或回改详细设计。

这三个检查点的作用不是礼貌征询，而是权限控制：

- agent 不能私自推进流程。
- 用户保留节奏和成本控制权。
- 复杂任务可以被迫停在设计阶段，不会被“自动实现”。

## 7. `spec-leader` 的真实角色

从提示词看，`spec-leader` 更像项目经理加流程控制器，而不是技术实现者。

它被明确禁止：

- 直接写代码。
- 直接修改设计文档。
- 直接调试问题。
- 直接分析代码逻辑。
- 直接执行测试。

但它必须负责：

- 创建任务目录和 task name。
- 组织 sub-agent 执行。
- 在每一轮 sub-agent 返回后检查 artifact 是否完整。
- 发现不完整时，重新调用相同 sub-agent 并说明缺失项。
- 在关键阶段征求用户确认。

这体现了一种非常典型的“强 orchestration，弱 execution”设计。

## 8. `spec-requirement-analyser` 的机制意义

这个 agent 不是简单把需求写成文档，而是承担了两个上游关键职责：

1. 区分分析任务与开发任务。
2. 与用户对齐真实意图，而不是替用户脑补。

其关键机制包括：

- 先判断任务类型，再决定后续路径。
- 对开发任务必须研究代码现状，再与用户讨论需求细节。
- 可以问功能边界、约束、测试要求，但禁止问工作流阶段选择。
- 输出需求文档后必须立即结束，把控制权交回 `spec-leader`。

这说明 Qodercli 把“需求澄清”从“流程编排”中拆了出来，避免编排器兼任业务分析导致的角色污染。

## 9. 验证策略被单独建模的原因

`spec-verify-requirement` 是这套系统里很关键但容易被忽略的一个设计点。

它不直接测代码，而是在编码之前先定义：

- 是否需要 UT。
- 是否需要 IT。
- 是否需要 E2E。
- 哪些场景可自动执行。
- 哪些主动验证动作需要用户确认。

这一步的价值很高：

1. 让测试不再是编码后的补丁，而是设计输入。
2. 把“可验证性”前置，倒逼设计时考虑自动化路径。
3. 为 `spec-verifier` 提供机器可执行的验证计划，减少验证阶段自由发挥。

本质上，它把“Definition of Done”文档化了。

## 10. 设计层机制：HLD / LLD / Review 的组合拳

### 10.1 高层设计

`spec-hld-designer` 输出高层设计，重点是：

- 模块边界。
- 接口关系。
- 数据流。
- 文件变更清单。
- 自动化测试方案。

它被明确要求聚焦架构和逻辑流，避免写实现细节。这是在防止设计文档退化成伪代码堆积。

### 10.2 详细设计

`spec-lld-designer` 进一步把高层设计拆成原子任务，生成：

- `overview.md`
- `task-NN-xxx.md`

每个任务都需要包含：

- 文件路径。
- 依赖关系。
- 核心逻辑。
- 边界条件。
- 单元测试设计。
- Mock 方案。

这一步本质是把架构方案编译成可执行施工清单。

### 10.3 设计评审

`spec-design-reviewer` 的机制并不是普通评论，而是一个完整质量门：

- 必须逐项对照需求检查覆盖率。
- 必须检查是否过度设计。
- 必须一次性给出所有阻塞项和建议项。
- 必须以轮次文件输出，不能覆盖历史记录。

这里的关键思想是“设计评审不是辅助动作，而是正式状态迁移前的审批节点”。

## 11. 编码执行机制

### 11.1 普通编码模式

没有详细设计时，`spec-implementer` 直接根据高层设计编码。

### 11.2 任务化编码模式

有详细设计时，编码必须严格按照任务顺序进行。核心约束有三条：

1. 禁止并行编码。
2. `overview.md` 中的状态必须从 `pending` 更新到 `in_progress`，再到 `completed`。
3. 中断恢复时，从第一个非 `completed` 任务继续。

这套机制说明系统希望用 LLD 文档承担“轻量任务编排器”的职责，而不是临时用对话记忆跟踪开发进度。

## 12. 验证与修复闭环

`spec-verifier` 只做验证，不做修复，这是这套机制里最干净的一条边界。

它的职责是：

- 读取验证计划。
- 检测是否存在可执行测试。
- 按 UT、IT、E2E 顺序执行。
- 执行主动验证动作。
- 在失败时给出详细根因分析与修复建议。

但它被禁止：

- 修改代码。
- 调用 `spec-implementer`。
- 决定是否重试。

这就形成一个非常清晰的闭环：

1. `spec-leader` 调 verifier。
2. verifier 返回报告。
3. `spec-leader` 决定是否调 implementer 修复。
4. 修复后重新验证。
5. 超过上限再交还用户决定。

这避免了“验证 agent 一边测一边偷偷改代码”的失控情况。

## 13. Prompt 在系统中的真实作用

这些 prompt 不只是描述行为风格，而是在承担四种机制职能：

1. 权限声明。
2. 工作流约束。
3. 输出格式协议。
4. 质量门规则。

例如：

- “只能输出到某目录”是存储边界。
- “不得直接修改代码”是权限边界。
- “必须先读 requirements”是依赖顺序约束。
- “完成后立即返回 spec-leader”是控制权回收协议。

因此，这些 prompt 更接近“运行时治理规则”，而不是自然语言说明书。

## 14. Task 调度协议

`spec-leader` 使用 `Task` 工具调 sub-agent 时，prompt 里必须带：

- 需要加载的 skill 名称。
- 背景信息。
- 已有 artifacts 位置。
- 当前任务描述。
- 语言要求。

之所以这样设计，是因为每次 sub-agent 调用都是独立新会话，没有共享记忆。

这带来两个重要结论：

1. 调度质量主要取决于 prompt 封装是否完整。
2. artifact 路径和任务上下文是跨 agent 协作的事实标准接口。

## 15. 这套机制的工程价值

### 15.1 优点

1. 降低越权执行概率。
2. 强化可追溯性，所有中间决策都有文档落地。
3. 支持中断恢复，不强依赖长上下文。
4. 通过评审和验证回路抑制一次性大改带来的风险。
5. 更适合复杂任务，而不是只适合单文件修改。

### 15.2 代价

1. 流程偏重，简单任务会显得繁琐。
2. prompt 与 artifact 模板维护成本高。
3. 编排器一旦 prompt 设计不好，会出现空转、重试不充分或上下文遗漏。
4. 文档质量差时，后续 agent 会在错误输入上继续工作。

### 15.3 适用场景

最适合以下场景：

- 跨模块复杂开发。
- 需求不稳定但需要过程留痕。
- 需要中断恢复与分阶段确认。
- 希望把测试策略前置。
- 希望把设计评审纳入 AI 工作流。

不太适合以下场景：

- 极小改动。
- 一次性脚本处理。
- 用户只要快速答案，不要流程资产。

## 16. 对架构设计的直接启示

从这套机制反推，适配 Qodercli 的架构设计必须满足以下要求：

1. 设计文档必须可分阶段消费，而不是一份超长总文档。
2. 设计要能自然映射到 artifact 结构和任务分解结构。
3. 测试方案必须前置，并明确哪些部分可自动验证。
4. 模块边界必须清晰，否则 HLD 很难通过 review，LLD 也无法拆任务。
5. 实现路径必须支持串行执行与中途恢复。

简化地说，这套系统偏好“边界清晰、文档可执行、验证可自动化”的架构。

## 17. 主要风险点

### 17.1 流程风险

- 需求文档不清晰，后面所有阶段都会被污染。
- 评审只做形式检查，不做逐条覆盖验证，会失去质量门意义。
- 跳过设计直接编码时，验证成本会上升。

### 17.2 组织风险

- 如果让编排器兼做实现，系统边界会快速崩溃。
- 如果 verifier 能自动修代码，会形成不可审计修改链。
- 如果没有 artifact 验证，sub-agent 结果不完整也会被推进到下一阶段。

### 17.3 设计风险

- 高层设计写成实现细节，会导致 LLD 失去价值。
- 任务拆分过粗，编码阶段难以控制风险。
- 任务拆分过细，流程摩擦过高，交付效率下降。

## 18. 建议的理解框架

理解这套机制时，建议不要按“agent 名称”记忆，而要按下面四层记忆：

1. 路由层：谁决定下一步找谁。
2. 状态层：当前处于哪个阶段，下一步允许去哪里。
3. 产物层：本阶段必须生成什么文档。
4. 权限层：谁能读什么、写什么、执行什么。

这样更容易把 Qodercli 看成一个“受 prompt 驱动的工程状态机”，而不是一组散乱的 agent 提示词。

## 19. 最终判断

Qodercli 的 multi-agent 机制是一个偏工程治理导向的设计，而不是偏开放探索导向的设计。

它最重要的创新点不在于 agent 数量，而在于三件事：

1. 用 prompt 明确职责边界。
2. 用 artifact 串联阶段状态。
3. 用 review 和 verify 回路把错误尽量挡在阶段边界上。

如果把它抽象成一句话，可以表述为：

“Qodercli 通过 `spec-leader` 驱动的文档化状态机，把需求、设计、实现、验证拆成受限 agent 工序，从而在 AI 参与软件工程时换取更高的可控性、可审计性和可恢复性。”
